# DATA SHEET — CommitDistill: A Lightweight Knowledge-Centric Memory Layer for Software Repositories

_Source PDF: `commitdistill-2605.18284.pdf` · extracted from the PDF, not HTML. Verify each quote at its cited location._

**Authors/venue:** Divya Chukkapalli, Thejesh Avula, Aditya Aggarwal, Harsimran Singh, Amith Tallanki. Microsoft Corporation. arXiv:2605.18284v1 [cs.SE] 18 May 2026. (Venue not stated beyond arXiv; "anonymised repository link to be inserted on acceptance.")

**What it does:** CommitDistill is an open-source pure-Python prototype (~510 lines core + ~80 lines tests, no third-party dependencies) that mines a local git history into typed knowledge units of three kinds (Facts/constraints, Skills/actionable solutions, Patterns/recurring failure modes) using deterministic regex heuristics (Algorithm 1: 9 regex patterns, 3 per type), stores them in a plain-JSON inspectable store (.knowledge/units.json), and retrieves them via a length-normalised TF-IDF retriever with per-type and prior boosting (Algorithm 2) plus a calibrated silence threshold (theta=2.5) that abstains on out-of-corpus / out-of-distribution queries. Quote (Abstract): "We present CommitDistill, an open-source Python prototype that mines a local git history into typed knowledge units (Facts, Skills, Patterns) using deterministic regex heuristics and surfaces them through a TF-IDF retriever with an empirically calibrated silence threshold (theta = 2.5) that abstains on out-of-distribution queries." No embeddings, no external service, local-only execution.

**Q: The HARD-stratum (control <= 0.5) lift numbers, exact, with table reference.**

**Q: n = ? number of tasks / bug-fixes evaluated.**

**Q: The extractor: regex / TF-IDF / rule-based? Description and yield/silence statistics.**
- Answer: The extractor is a deterministic, regex-driven pipeline (Algorithm 1), 9 regular-expression heuristics (3 per unit type), not TF-IDF and not LLM-based. (TF-IDF is used only in the retriever, Algorithm 2.) Silence statistics: on the n=200 corpus the original CD-v1 (regex-only) returned ZERO substantive units on 96/100 requests fixes and 100/100 flask fixes (~98% silence). CD-v2's subject-fallback unit reduces extractor silence from 98% to 34.5%. Time-travel coverage: only 1/40 fixes had any CD-v1 top-10 unit in ground truth G, vs 7/40 grep and 12/40 BM25.
- Quote: "The extractor is a deterministic, regex-driven pipeline. Each heuristic pattern is associated with a unit type and a prior weight; matched spans are normalised, length-filtered, and deduplicated by a content-derived hash. ... Algorithm 1 returned zero substantive units on 96/100 requests fixes and 100/100 flask fixes (i.e., ~98% silence)"
- Location: Section IV-B (Extraction, p.4); Section VI-G2 (p.10) for the 98%->34.5% silence figures; Fig.2 lists the patterns

**Q: What is the evaluation TASK, the grader/judge, and any inter-rater agreement (kappa)?**
- Answer: Tasks: (RQ5) time-travel bug-fix file/symbol localisation — the LLM predicts up to 5 files and up to 5 identifiers most likely to need editing; (RQ4) regression-finding: recover prior co-changing bug-fix commits (Hit@1/3/10, MRR); (RQ3) retrieval relevance. Grader for RQ5: two independent LLM judges, GPT-4o and GPT-4o-mini, scoring on a 0-2 rubric (0=not useful, 1=partially useful, 2=useful). RQ2 grader: two co-author human annotators (four-class rubric useful/trivially-true/fragment/noise). Kappa: human RQ2 Cohen's kappa = 0.633 (substantial agreement, 31/40 labels agreed). LLM judges: Cohen's kappa = 0.355 (four-arm) / 0.366 (three-arm slice), 'fair' band.
- Quote: "we ran a downstream LLM-as-judge experiment on n=200 time-travel bug-fix commits ... the assistant LLM (GPT-4O-MINI) only the cleaned bug subject and the project name, and asked it to predict (a) up to five files most likely to need editing and (b) up to five identifier names most likely involved. ... Two independent judge models (GPT-4O and GPT-4O-MINI) scored each assistant answer on a 0-2 rubric ... Cohen's kappa between GPT-4O and GPT-4O-MINI is 0.355 ... Cohen's kappa = 0.633 on the human RQ2 rubric."
- Location: Section VI-G (RQ5, p.9); Section VI-B (RQ2, p.7) for kappa=0.633; Section VI-G2 (p.10) for kappa=0.355

**Q: The temporal / leakage-control protocol (time-travel split? memory built before fix date?).**
- Answer: Yes — a time-travel protocol. For RQ4/RQ5, the retriever's view of the world is restricted to commits strictly before the fix; the fix's own commit is excluded from rankings. The retriever's state is built from commits made strictly before the target fix's author date. For Table VII the world is restricted to the 5,000 commits strictly before the fix.
- Quote: "we build the retriever's state from commits made strictly before that fix's author date, use the cleaned commit subject as the query, and ask each retriever to return ten candidates. ... For each fix, the retriever's view of the world is restricted to the 5,000 commits strictly before the fix; the fix's own commit is excluded from rankings."
- Location: Section VI-F (RQ4, p.9) and Table VII caption (p.9)

## Evaluation setup
Illustrative case study on five public repositories: psf/requests (Python), pallets/flask (Python), expressjs/express (JavaScript), redis/redis (C), junit-team/junit5 (Java). Extractor run over most recent 5,000 commits/subject for yield (25,000 total) and most recent 2,000/subject (10,000 total) for the retrieval evaluation. Five research questions: RQ1 yield, RQ2 precision (n=40 dual-annotated units from requests n=24 + flask n=16, two co-author human annotators, four-class rubric), RQ3 retrieval (36 queries = 6 hand-authored + 30 auto-derived; baselines git log -i --grep and pure-stdlib BM25 over raw commit subjects+bodies), RQ4 time-travel regression-finding (40 bug-fix commits, 20 requests + 20 flask; ground truth = prior co-changing bug-fix commits; metrics Hit@1/3/10, MRR; retriever restricted to commits strictly before the fix), RQ5 downstream LLM-as-judge (n=200 time-travel bug-fix file/symbol localisation, 100 requests + 100 flask; assistant LLM GPT-4o-mini predicts up to 5 files + 5 identifiers; four arms = Control/subject-only, +CommitDistill CD-v2, +CD-Hybrid, +BM25 top-3; two judge models GPT-4o and GPT-4o-mini, 0-2 rubric; paired bootstrap 95% CIs, 10,000 resamples). Budget-constrained retrieval: 12 hand-curated fact-style queries, budget B in {64,128,256,512,1024,2048} chars, top-10 greedily packed. Silence threshold theta calibrated on 15 queries partitioned into ANSWERABLE (n=3), NOT_IN_CORPUS (n=3), OOD (n=6); default theta=2.5. Hardware: Intel Core i7-1185G7 laptop, 4 cores/8 threads, 16GB RAM, Windows 11. Paper deliberately reports NO significance tests on Tables II-IV because sample sizes do not support them.

## Stated limitations
Section VIII (Threats to Validity, following Wohlin et al.) and Section VII-B. (1) Construct validity: "Useful" depends on annotator judgement; rubric is the authors' and may not match working developers; LLM-judge kappa weaker (0.355) driven by absolute-scale calibration. (2) Trustworthy-AI threat model: commit-prose poisoning (no authentication beyond commit-author metadata; a malicious contributor can write a deceptive unit), privacy (distilled store may verbatim-preserve internal commit fragments), confidentiality of attribution, adversarial silence circumvention. (3) Internal validity: annotators were co-authors aware of the system's intent (a fully blinded protocol would be stronger, planned as follow-up). (4) External validity: precision measured only on two Python subjects (n=40); yield depends on commit-message culture (redis 329.5 units/kc vs flask 8.0 — order-of-magnitude gap); 36 queries too few to characterise retrieval recall in general; auto-derived queries trivially favour the baselines; RQ5 uses two OpenAI-family judges. (5) Conclusion validity: deliberately no significance tests on Tables II-IV; intervals wide; RQ5 framed as "null with a slight negative trend for CommitDistill, not evidence of equivalence." Section VII-B "When CommitDistill Does Not Help": recall is the dominant limitation; yield depends on commit-message culture; the retriever is lexical (cannot bridge "redirect loop" to "infinite 302 chain"). Time-travel task (RQ4) is an out-of-design stress test the system was NOT built for and where it is "structurally outperformed by BM25 over raw commits." The four-arm ablation (CD-v2/CD-Hybrid) was designed AFTER observing the three-arm result, so Delta-values are exploratory, not confirmatory (flagged as a construct-validity threat).

## NOT stated in paper
The paper does NOT claim statistical superiority over any existing tool, does NOT benchmark against PlugMem (explicitly: "we do not benchmark against PlugMem"), and does NOT claim it competes with or replaces neural/embedding retrievers or modern coding assistants. It does NOT establish a positive end-to-end LLM patch-generation utility claim ("RQ5 does not establish a positive mean-utility claim for CommitDistill ... and we make none"). It does NOT demonstrate the distillation-helps-the-LLM hypothesis (stated as falsifiable but unproven). It does NOT claim precision generalises beyond the two Python subjects. It does NOT claim equivalence between typed units and raw prose (only "statistically indistinguishable" with wide CIs spanning zero). No human-subjects user study was run (planned as future work); no IRB approval required. No held-out replication on a third repository was performed (planned, e.g., httpx or urllib3). The four-arm deltas are explicitly exploratory, not pre-registered/confirmatory.
