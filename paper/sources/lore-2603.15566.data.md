# DATA SHEET — Lore: Repurposing Git Commit Messages as a Structured Knowledge Protocol for AI Coding Agents

_Source PDF: `lore-2603.15566.pdf` · extracted from the PDF, not HTML. Verify each quote at its cited location._

**Authors/venue:** Ivan Stetsenko, Independent Researcher (dev.ivan.stetsenko@gmail.com). Preprint, arXiv:2603.15566v1 [cs.SE], 16 Mar 2026. Title page also dated "March 2026". Single author.

**What it does:** The paper proposes "Lore," a lightweight protocol that restructures git commit messages — using native git trailers — into self-contained "decision records" (called "Lore atoms") carrying constraints, rejected alternatives, agent directives, and verification metadata. Per the abstract: "I propose Lore, a lightweight protocol that restructures commit messages—using native git trailers—into self-contained decision records carrying constraints, rejected alternatives, agent directives, and verification metadata. Lore requires no infrastructure beyond git, is queryable via a standalone CLI tool, and is discoverable by any agent capable of running shell commands. This paper formalizes the protocol, compares it against five competing approaches, stress-tests it against its strongest objections, and outlines an empirical validation path." A commit's intent line describes WHY (not what changed); the body gives narrative context; trailers (Constraint:, Rejected:, Confidence:, Scope-risk:, Reversibility:, Directive:, Tested:, Not-tested:, Related:) carry machine-parseable decision context (Table 1). A CLI tool (lore) provides scoped querying (context, constraints, rejected, directives, coverage, stale) and authoring (commit, commit --from-json, validate). The CLI is optional; the trailer format alone is sufficient via "git log --trailer=". The paper states it "formalizes the protocol" and "outlines an empirical validation path" — it does not implement or run an experiment.

## Load-bearing findings

**Q: Does Lore work ONLY on NEW commits going forward (forward-only authoring), or does it ALSO retroactively extract/distill from a repository's EXISTING git history?**
- Answer: Forward-only authoring. The paper describes agents serializing decision context at commit time into NEW commits; it does NOT describe any retroactive extraction or distillation from existing git history. The validate command only checks 'recent commits'. No backfill/mining of existing history is mentioned.
- Quote: "Knowledge serialization. The agent formats its own commit as a Lore atom, serializing decision context it already holds rather than discarding it. ... lore validate  Check recent commits for lore format"
- Location: Section 3.4 Agent Consumption Model (p.5) and Figure 2 Lore CLI interface (p.6)

**Q: What exactly does its validate (and doctor) command check?**
- Answer: There is a `lore validate` command described only as checking recent commits for lore format compliance. There is NO 'doctor' command in the paper. The paper does not describe validate checking semantic contradiction between records or contradiction against current code/repo state — only format compliance. (A separate `lore stale` command flags possibly-outdated constraints/assumptions, but does not check contradiction against code.)
- Quote: "lore validate          Check recent commits for lore format"
- Location: Figure 2 Lore CLI interface (p.6); also Section 5 'rubber-stamped low-quality atoms require lore validate checks'; 'doctor' is not stated in paper

## Evaluation setup
No empirical evaluation was conducted. The paper explicitly only PROPOSES an empirical validation: "Proposed empirical validation: Two teams, same project, six months. One uses Lore; one uses conventional commits. Measured outcomes: agent task success rate, time-to-correct-solution, rate of re-proposing rejected approaches, and review cycles before merge." (Section 7, p.7). No benchmarks, no datasets, no N, no baselines run, no agent scaffold executed. The paper's stated method of evaluation is conceptual: it "formalizes the protocol, compares it against five competing approaches [ADRs, Code Comments, AI-Generated Diff Summaries, RAG Over External Sources, Code Digital Twin — Section 4], stress-tests it against its strongest objections [Section 5 Critical Examination], and outlines an empirical validation path" (Abstract, p.1).

## Stated limitations
The paper states several concessions in Section 5 (Critical Examination, p.6-7): (1) Overhead/quality risk: "rubber-stamped low-quality atoms require lore validate checks." (2) Context window objection / scope: "Concession: Lore's value is primarily inter-session and inter-agent." (3) Trust/gaming: "Lore does not solve the trust problem; it makes trust auditable." (4) Granularity mismatch: "Concession: ADRs [4] remain better for cross-cutting architectural decisions." It also notes ADRs are complementary (implementation-level vs architectural granularity). In the Conclusion it frames its minimalism as "a feature, not a limitation." Disclosure: AI-assisted tools (Claude — Anthropic, claude-opus-4-6; Google NotebookLM) were used in research and writing.

## NOT stated in paper
- No retroactive extraction/distillation/mining of existing git history is described; Lore is forward authoring of new commits only. - No empirical/experimental results, no agent task-success numbers, no run study (only a proposed one). - No 'doctor' command exists. - validate is described only as a format/compliance check on recent commits; it does NOT check semantic contradiction between records, nor contradiction against current code/repo state. - No retrieval-method details such as BM25, embeddings, or vector search are mentioned. - CommitDistill, ReasoningBank, and BM25 are NOT cited. - No reference implementation, no code repository, and no released dataset for the CLI tool are provided. - No statistical results, ablations, or baselines are reported.
