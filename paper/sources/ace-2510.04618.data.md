# DATA SHEET — Agentic Context Engineering: Evolving Contexts for Self-Improving Language Models

Equal contribution noted for the first two authors.

**What it does:** ACE (Agentic Context Engineering) is a framework for context adaptation in both offline (e.g., system prompt optimization) and online (e.g., test-time memory adaptation) settings. Per the paper: "ACE treats contexts as evolving playbooks that accumulate and organize strategies over time." It uses a modular workflow of three roles (Figure 4): a Generator that produces reasoning trajectories; a Reflector that "distills concrete insights from successes and errors"; and a Curator that "integrates these insights into structured context updates." Contexts are represented as "structured, itemized bullets," each with metadata (a unique identifier and counters tracking how often it was marked helpful or harmful) and content. Instead of monolithic rewrites, ACE produces "compact delta contexts" (incremental delta updates) merged deterministically by non-LLM logic, plus a "grow-and-refine" mechanism that appends new bullets, updates existing ones in place (incrementing counters), and de-duplicates redundant bullets by comparing them via semantic embeddings. The paper states ACE can adapt "without labeled supervision and instead by leveraging natural execution feedback."

**Q: What are the method components (Generator / Reflector / Curator)?**
- Answer: ACE introduces three roles: Generator (produces reasoning trajectories), Reflector (distills concrete insights from successes and errors), and Curator (integrates these insights into structured context updates). Three key innovations: (1) a dedicated Reflector separating evaluation/insight extraction from curation; (2) incremental delta updates replacing monolithic rewrites; (3) a grow-and-refine mechanism.
- Quote: "ACE introduces a structured division of labor across three roles (Figure 4): the Generator, which produces reasoning trajectories; the Reflector, which distills concrete insights from successes and errors; and the Curator, which integrates these insights into structured context updates."
- Location: Section 3 (Agentic Context Engineering), page 4

**Q: What are delta updates?**
- Answer: Incremental delta updates replace costly monolithic rewrites with localized edits. The Curator synthesizes lessons into compact delta entries that are merged deterministically into the existing context by lightweight, non-LLM logic; because updates are itemized and localized, multiple deltas can be merged in parallel.
- Quote: "(2) incremental delta updates (§3.1) that replace costly monolithic rewrites with localized edits, reducing both latency and compute cost ... The Curator then synthesizes these lessons into compact delta entries, which are merged deterministically into the existing context by lightweight, non-LLM logic. Because updates are itemized and localized, multiple deltas can be merged in parallel, enabling batched adaptation at scale."
- Location: Section 3 and Section 3.1, pages 4-5

**Q: What is grow-and-refine?**
- Answer: A mechanism that keeps contexts compact and relevant via periodic or lazy refinement: bullets with new identifiers are appended while existing bullets are updated in place (e.g., incrementing counters), and a de-duplication step prunes redundancy by comparing bullets via semantic embeddings. Can be performed proactively (after each delta) or lazily (when context window exceeded).
- Quote: "In grow-and-refine, bullets with new identifiers are appended, while existing bullets are updated in place (e.g., incrementing counters). A de-duplication step then prunes redundancy by comparing bullets via semantic embeddings."
- Location: Section 3.2 (Grow-and-Refine), page 5

**Q: What is the memory structure (itemized bullets with helpful/harmful counters)?**
- Answer: Yes. Context is represented as a collection of structured, itemized bullets; each bullet consists of (1) metadata, including a unique identifier and counters tracking how often it was marked helpful or harmful, and (2) content capturing a small unit such as a reusable strategy, domain concept, or common failure mode.
- Quote: "each bullet ... consists of (1) metadata, including a unique identifier and counters tracking how often it was marked helpful or harmful; and (2) content, capturing a small unit such as a reusable strategy, domain concept, or common failure mode. When solving new problems, the Generator highlights which bullets were useful or misleading, providing feedback that guides the Reflector in proposing corrective updates."
- Location: Section 3.1 (Incremental Delta Updates), page 5

**Q: What baselines are used (ICL, GEPA, MIPROv2, Dynamic Cheatsheet)?**
- Answer: Baselines: Base LLM; In-Context Learning (ICL); MIPROv2 (DSPy implementation, auto="heavy"); GEPA (DSPy implementation, auto="heavy"); Dynamic Cheatsheet (DC) in cumulative mode (DC-CU). For AppWorld, baselines are built on the official ReAct implementation.
- Quote: "In-Context Learning (ICL) ... MIPROv2 ... GEPA (Genetic-Pareto) ... Dynamic Cheatsheet (DC) ... We use the official implementation released by the authors (Suzgun et al.) and set it to use the cumulative mode (DC-CU)."
- Location: Section 4.2 (Baselines and Methods), pages 6-7

**Q: Does ACE use any consistency/verification check on its memory? State exactly what, if anything.**
- Answer: ACE does not perform a self-consistency or correctness-verification step on memory contents in its core method. Its only memory-hygiene mechanisms are: grow-and-refine de-duplication via semantic embeddings, and helpful/harmful counter metadata used to flag bullets. The paper explicitly lists contradiction detection and prioritizing high-confidence updates as 'compatible extensions that could further improve' the system — i.e., NOT part of ACE as implemented. The Reflector can optionally refine insights across multiple iterations ('iterative refinement'), but this is insight distillation, not a consistency check on stored memory.
- Quote: "ACE's bullet-point analyzer ("grow-and-refine"; §3.2), which merges and deduplicates semantically similar bullets and can filter entries flagged as potentially harmful via metadata, serves as a first line of defense against context noise. Additional safeguards (e.g., contradiction detection, prompting the Curator to prioritize high-confidence updates, or periodic pruning of outdated entries) are compatible extensions that could further improve playbook compactness and consistency."
- Location: Section 3.2 (page 5) for grow-and-refine; Appendix A.4 'Takeaway and Mitigation' (page 19) for the explicit statement that contradiction detection is an extension, not implemented

- Answer: No, ACE can operate without ground-truth labels by leveraging execution feedback and environment signals. The paper evaluates both 'with GT labels' and 'without GT labels' variants (shown by checkmark / X in the GT Labels column of Tables 1, 2). - Quote: "ACE is able to construct effective contexts without labeled supervision, instead leveraging execution feedback and environment signals, key ingredients for self-improving LLMs and agents."

## Evaluation setup
Two categories evaluated. (1) LLM Agent: AppWorld (Trivedi et al., 2024) — autonomous agent tasks with API understanding, code generation, environment interaction; difficulty levels normal and challenge; metrics Task Goal Completion (TGC) and Scenario Goal Completion (SGC) on test-normal and test-challenge, following official protocol. Built on the official ReAct implementation. (2) Domain-specific reasoning (finance main case study): FiNER (Loukas et al., 2022) — labeling tokens in XBRL financial documents with one of 139 fine-grained entity types; Formula (Wang et al., 2025a) — financial numerical reasoning. Additional domains from StreamBench: DDXPlus (medical reasoning) and BIRD-SQL (text-to-SQL). Metrics: accuracy as exact match for FiNER/Formula/DDXPlus; BIRD-SQL uses GPT-4o-mini as LLM-as-a-judge. Datasets follow original train/validation/test splits. Offline adaptation: methods optimized on training split, evaluated on test split with pass@1; online adaptation: methods evaluated sequentially on the test split (predict with current context, then update). Default backbone: DeepSeek-V3.1 (non-thinking mode); same LLM used for Generator, Reflector, and Curator to ensure fairness. Batch size 1 (one delta context per sample). Maximum Reflector refinement rounds and maximum offline epochs both set to 5. Baselines: Base LLM, ICL, MIPROv2 (DSPy, auto=heavy), GEPA (DSPy, auto=heavy), Dynamic Cheatsheet (DC-CU cumulative mode). Generalization across LLMs tested with GPT-OSS-120B, GPT-5.1, and Llama-3.3-70B-Instruct (Appendix A.1, Tables 5-9). Cost analysis vs GEPA/DC in Tables 4, 12-15. Robustness tests on reflector quality in Appendix A.4 (Tables 16-17). Ablations in Table 3 and Table 18.

## Stated limitations
The paper (Section 5, Limitations and Challenges, page 10) states: "A limitation of ACE is its reliance on a reasonably strong Reflector: if the Reflector fails to extract meaningful insights from generated traces or outcomes, the constructed context may become noisy or even harmful. In domain-specific tasks where no model can extract useful insights, the resulting context will naturally lack them. This dependency is similar to Dynamic Cheatsheet ... where the quality of adaptation hinges on the underlying model's ability to curate memory." Also: "not all applications require rich or detailed contexts. Tasks like HotPotQA often benefit more from concise, high-level instructions ... Similarly, games with fixed strategies such as Game of 24 may only need a single reusable rule, rendering additional context redundant." In Section 4.4 the paper also notes that when ground-truth supervision or reliable execution signals are absent, both ACE and DC may degrade in performance, as the context "can be polluted by spurious or misleading signals, highlighting a potential limitation of inference-time adaptation without reliable feedback." Appendix A.4 notes performance drops below the base model only under sustained adversarial corruption (harmful reflector every iteration).

## NOT stated in paper
The paper does NOT claim ACE performs any explicit self-consistency voting, contradiction-detection, or correctness-verification step on its stored memory bullets — contradiction detection is explicitly described as a "compatible extension that could further improve" the system, not part of the implemented method (Appendix A.4). The paper does NOT use the AppWorld leaderboard top agent IBM CUGA as a methodological baseline (footnote 1 states it is "a rough contextual reference ... not used as a methodological baseline, and we do not make direct comparisons"). The paper does NOT claim ACE always improves without ground-truth labels — in several settings (e.g., FiNER online without GT, Formula online DC) accuracy degrades below base. The paper does NOT compare ACE to MCTS or tree-search-based memory methods (no such comparison stated). The paper does NOT claim grow-and-refine de-duplication uses anything other than semantic embeddings. The paper does NOT report human evaluation. The exact embedding model used for semantic de-duplication is not stated in the read sections. Precise dataset sizes (N) for FiNER/Formula/AppWorld test splits are not all stated in main text (DDXPlus 1372 train/val and BIRD-SQL 372 examples mentioned in Appendix A.2; 1000 training examples used for offline finance adaptation).
