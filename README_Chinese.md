# MCTS-Mem

### 记录代码库背后的设计理由

代码能说明项目现在怎样运行，Git 历史能说明它一路改过什么。

但两者往往都回答不了一个问题：项目为什么会形成今天的设计？

例如，一个项目目前使用 QuickJS。从代码可以确认当前用的是 QuickJS，却未必看得出团队为什么不再使用 V8：是体积太大、运行行为不符合预期，还是维护成本太高？项目是否试过其他方案？当时的限制现在还成立吗？

MCTS-Mem 用一棵结构化的设计树记录这些信息：

- 目前采用的方案；
- 它所取代的方案；
- 选择当前方案所依据的事实；
- 每条事实的来源。

这棵树由 Markdown 文件组成，与代码放在一起，并由检查工具（linter）核对内部结构是否一致。下文从一个具体例子开始，说明什么是设计记忆、目录和文件怎样组织，以及如何为现有项目建立并维护这棵树。

## 1. 什么是设计记忆

假设一个渲染引擎现在通过 QuickJS 执行 JavaScript。从代码中可以找到 QuickJS 的实现，却不一定看得到下面这段历史：

1. 项目最初使用 V8；
2. 集成 V8 会让应用包增大太多，因此团队换掉了它；
3. 团队随后试过 JavaScriptCore；
4. 引擎还要在 Apple 平台以外运行，因此 JavaScriptCore 也被换掉了。

这段缺失的信息就是项目的设计记忆。

```text
exec-engine ▸ quickjs
  .alt/ v8  ✗ 包体积太大
  .alt/ jsc ✗ 仅支持 Apple 平台
```

设计记忆既保存最后采用的方案，也保存已经放弃的分支。

这一区别很重要。一个被否决的设计在技术上仍可能显得可行。如果否决原因没有留下记录，新加入的工程师或编程智能体很容易再次提出同一个方案。

## 2. 设计理由为什么会丢失

一个项目里，通常同时存在两类信息。

一类是实现本身。Git 记录它怎样变化，测试验证它的行为，编译器则能发现其中许多不一致。

另一类是实现背后的理由。这些内容往往散落在不同地方：

- 设计文档和 RFC；
- issue 和 pull request；
- 提交说明；
- 基准测试记录；
- 聊天记录；
- 代码注释；
- 决策者的记忆。

问题在于，这些设计理由没有对应的“编译器”。

一份设计文档可以继续描述两年前已经删除的实现；两份 RFC 即使相互矛盾，也不会触发错误；某次基准测试的数据后来得到修正，Wiki 中基于旧数据写下的决策却可能一直没有更新。

即使这些解释逐渐与代码脱节，构建和测试仍可正常通过。因此，过时的理由往往看起来和仍然有效的理由一样可信。

还有一类设计理由从一开始就没有被写下来。一条架构约定可能只作为团队习惯存在；一项限制可能只因为某位工程师还记得而继续生效；一条需求可能停留在聊天记录里，没有变成任何可检查的约束。这类未被记录的意图相当于一种债务：项目依赖它，却没有文档记录它，也没有检查能强制它成立。

### 2.1 按时间记录决策有什么局限

架构决策记录（Architecture Decision Record，简称 ADR）是现有格式中与 MCTS-Mem 最接近的一种。ADR 通常记录一次决策的背景、内容和后果。项目第一次作出选择时，这种形式很有效。

如果同一个设计问题经历了多次选型，这些记录就会变得难以阅读。

仍以执行引擎为例。假设它依次经历了三种实现：

```text
V8 → JavaScriptCore → QuickJS
```

相关说明可能分散在多年间陆续写成的三份 ADR 中。要弄清当前方案，读者需要：

1. 找到全部三份记录；
2. 理清它们的先后顺序；
3. 判断哪些说法现在仍然成立；
4. 找出每个备选方案被放弃的原因。

MCTS-Mem 不以日期为主线，而是围绕设计问题组织这些信息。三种方案都放在“执行引擎”节点下。

V8、JavaScriptCore 和 QuickJS 都出现在这个节点附近。带日期的记录仍会保留演变顺序，但读者不必再从按时间排列的日志中自行拼出它们之间的关系。

### 2.2 按相似度检索有什么局限

另一种办法是把项目文档全部放进检索系统，再让大语言模型查找相关段落。这种方法很有用，但它解决的是另一个问题。

大多数检索增强生成系统（Retrieval-Augmented Generation，简称 RAG）按文本相似度返回内容。但与查询最相似的段落，不一定包含选择当前方案的理由。

例如，搜索“V8”可能得到：

- V8 构建说明；
- V8 绑定代码；
- V8 堆配置；
- 旧的 V8 调试笔记。

这些内容可能都已经过时。真正重要的事实也许是：

> QuickJS 取代原有引擎后，应用包体积减少了 40 MB。

这句话未必会在“V8”查询中排到前面。

对于 zlib 这类常见依赖，问题恰好相反。搜索“zlib”可能返回几十处调用点，却把真正说明项目为何选择它的那条记录埋在后面。

```text
RAG：按相似文本查找                    MCTS-Mem：按决策查找

查询 "V8"                            exec-engine ▸ quickjs
  V8 绑定代码                            .alt/ v8  ✗ 包体积
  V8 构建选项                            .alt/ jsc ✗ 平台支持
  V8 堆调优                              事实、依赖和变更记录

查询 "zlib"                          compression ▸ zlib
  图像解码器                             .alt/ miniz ✗ 压缩率较差
  字体加载器
  资源打包器
  网络缓存
  ...另外 36 处调用
```

RAG 可以帮助找到原始材料。确定这些材料对应哪项设计选择后，MCTS-Mem 再把它们组织起来。

## 3. 把代码库看成一次设计搜索

MCTS-Mem 这个名字来自一个类比：软件开发可以看成在多个候选设计之间搜索。

团队先确定要解决的问题，再尝试一种方案。把方案实现到足以验证的程度后，团队会从测试、基准评测、移植失败、线上事故或代码评审中得到反馈，再据此保留或否决方案，并决定下一步尝试什么。

这个过程与蒙特卡洛树搜索（Monte Carlo Tree Search，简称 MCTS）的四个常见步骤相似：

1. **选择（Select）**一个值得研究的设计问题。
2. **扩展（Expand）**一个候选方案。
3. **推演（Roll out）**这个候选方案，也就是实现并评估它。
4. **回传（Backpropagate）**得到的结果，让后续选型可以利用这些信息。

两者的对应关系如下。

```text
MCTS                                  软件开发

选择一个节点                           选择一个设计问题
扩展一个选项                           实现一种方案
运行一次模拟                           构建、测试、基准评测或评审
回传结果                               记录项目得到的结论
```

MCTS-Mem 并不会在代码仓库上运行传统的 MCTS 算法。这个名字借用的是设计过程的树状形态，以及搜索过程中需要保留的信息。

在棋盘游戏中，程序可以低成本运行数百万次模拟。在软件开发中，一次有意义的推演可能需要数天的开发工作、一次硬件移植或一整轮基准评测。因此，软件项目很难进行大量推演，单次成本也很高。

如果团队完成一次代价很高的集成后才发现 V8 不合适，后来者不应该再做一遍同样的集成，只为重新发现同一个限制。MCTS-Mem 保存的正是这段设计探索的历史。

实际上，这段设计探索往往已经保存在一个人的记忆里，也就是有经验的架构师。他们的价值很大程度上来自在多个项目中积累的经验，而每一条经验都是一次过往搜索的结果，记录着哪条路走得通、哪条走不通。这种记忆相当于一棵非正式的 MCTS 树，能让资深工程师避开新人会再次踩中的高成本弯路。但它也会随着人的离开而流失。把这段搜索写下来，就把个人记忆变成了项目可以长期保留、并供智能体查阅的资产。

这也让人看清额外的能力用在哪里更有效。更强的模型能更快地搜索设计空间；而一段已经记录下来的搜索历史，则从一开始就减少了需要搜索的量，因为每个被否决的分支都已经带着它失败的原因。

## 4. 设计树怎样表示历史

MCTS-Mem 设计树是一个由 Markdown 文件构成的目录。每个节点记录一项设计选择。

- **当前节点（live node）**记录当前仍然采用的方案；
- 同级的 **`.alt/` 目录**保存已经否决或被取代的方案；
- **Facts** 记录观察结果和理由；
- **Moves** 记录 `replaced`、`replaced by`、`dropped`、`revived` 等方案变更；
- **`.fact/` 目录**可以存放篇幅太长、不适合写进短条目的补充材料。

忽略所有 `.alt/` 目录，剩下的树就是项目的当前设计。查看 `.alt/` 目录，则能看到围绕同一问题尝试过、但最终没有采用的方案。

### 4.1 一个小例子

假设某个渲染引擎最初用 Skia 做光栅化，用 V8 执行 JavaScript。后来，团队两次更换 JavaScript 引擎，又更换了一次光栅化实现。

```text
S0：最初版本                  S1：重选执行引擎              S2：重选光栅化实现

render-engine                 render-engine                 render-engine
├─ dsl                         ├─ dsl                         ├─ dsl
├─ dsl-parser                  ├─ dsl-parser                  ├─ dsl-parser
├─ renderer                    ├─ renderer                    ├─ renderer
│  └─ rasterizer = skia       │  └─ rasterizer = skia       │  └─ rasterizer ▸ skity
└─ exec-engine ▸ v8            └─ exec-engine ▸ quickjs      │     .alt/ skia ✗ 复杂度
                                 .alt/ v8  ✗ 包体积        └─ exec-engine ▸ quickjs
                                 .alt/ jsc ✗ 平台支持          .alt/ v8  ✗ 包体积
                                                                 .alt/ jsc ✗ 平台支持
```

这些备选方案都平铺在同一层级。V8 和 JavaScriptCore 虽然是在不同时间采用的，但都是当前执行引擎的同级备选项。项目采用这些方案的先后顺序记录在带日期的 `Moves` 中。

这个例子只用于说明格式。Lynx 确实使用 Skity 作为光栅化器，例子中的其余历史则是虚构的。

### 4.2 文件内容

上图可以对应到下面的目录：

```text
mcts_mem/
  render-engine.md
  render-engine/
    exec-engine.md
    exec-engine.alt/
      v8.md
      jsc.md
```

当前采用的 `exec-engine.md` 节点可以这样写：

```md
- JavaScript 通过 QuickJS 运行。

## Facts

- 2026-05-12 rationale: QuickJS 支持所有目标平台，应用包体积也符合预算。(sourced)

## Moves

- 2026-02-08 (ab12cd34) replaced [[v8]]: V8 构建让应用包增大了 40 MB。(code)
- 2026-05-12 (cd34ef56) replaced [[jsc]]: JavaScriptCore 不支持项目需要覆盖的非 Apple 平台。(sourced)
```

冻结后的 V8 备选节点对同一替换关系作如下记录：

```md
- JavaScript 通过 V8 运行。

## Moves

- 2026-02-08 (ab12cd34) replaced by [[exec-engine]]: V8 构建让应用包增大了 40 MB。(code)
```

当前节点记录 QuickJS 取代了 V8；V8 的备选节点则记录自己被当前方案取代。两处替换记录中冒号后的原因必须逐字相同。如果两者不一致，linter 会报错。

## 5. 事实来源与不确定性

每条 Fact 和 Move 都必须以三个来源标记之一结尾：

- `(code)`：这项说法可以用代码、测试、二进制文件或其他项目产物核验；
- `(sourced)`：这项说法可以在提交记录、issue、文档、论文、聊天记录或作者陈述中找到依据；
- `(uncertain)`：现有资料中没有更有力的证据支持这项解释。

标签只说明证据来自哪里，不保证说法一定正确。

带 `(sourced)` 的记录本身可能有误；带 `(code)` 的基准测试也可能选错工作负载。但来源标记能让后来的读者知道现有证据属于哪一类。

恢复旧项目的设计历史时，这一区分尤其有用。Git 可能只显示团队把一种实现换成了另一种，却没有说明原因。MCTS-Mem 不会猜测一个看似合理的理由，而是如实记录这次替换，并把缺失的解释标为 `(uncertain)`。

例如：

```text
2021-08-14 rationale:
  项目替换了按端点分别维护的缓存，但现存记录没有说明原因。(uncertain)
```

如果后来找到一份旧 RFC，能够说明替换原因，就追加一条带日期、标记为 `(sourced)` 的 Fact。

同理，这些记录采用只追加（append-only）的方式维护。如果后来发现某次基准测试有误，应增加一条带日期的更正，不要为了让历史显得整齐而修改原始结果。

## 6. linter 检查什么

运行下面的命令可以检查一棵 MCTS-Mem 设计树：

```sh
npx mcts-mem lint mcts_mem
```

linter 会检查：

- 节点链接能否解析；
- 每次替换是否有相互对应的两条记录；
- 成对替换记录的 `reason` 是否一致；
- Fact 和 Move 是否带有来源标记；
- 已提交的历史是否遵守只追加规则；
- 条目是否符合预期的 Markdown 结构；
- 节点是否确实记录设计选择，而不是照着源码目录逐项抄写。

`lint` 没有报错，说明设计记忆符合上述结构规则，各条记录相互一致。但这不代表每项说法都正确。

linter 无法判断以下问题：基准测试的设计是否合理，旧文档是否可靠，团队从失败原型中得出的结论是否正确。这些问题仍需要工程判断。因此，linter 只检查设计理由的组织方式，不会把格式正确等同于结论正确。

## 7. 为现有项目建立设计树

为新项目建立设计树相对容易。但更常见的情况是，项目已经积累了多年历史，相关信息分散在代码、提交记录和文档里。

`mcts-mem-build` 技能按阶段重建这段历史：

```text
当前代码 + Git 历史 + 设计文档、RFC 和讨论
                         │
                         ▼
1. 根据当前代码建立骨架
2. 从提交记录中找出设计选择和被否决的方案
3. 把缺少理由的记录标为 (uncertain)
4. 用文档和讨论补回缺失的理由
5. 运行 lint 并审计结果
                         │
                         ▼
                  一棵通过 lint 的树
```

重建工作从当前代码开始，因为它代表项目现在真正采用的设计。

接下来从当前版本出发，回溯 Git 历史，找出项目重新选型的关键点，例如更换运行时、变更序列化方式、重写架构、放弃子系统，以及其他重要的设计分叉。

并非每次提交都要变成节点。格式调整、常规缺陷修复，以及处理方式基本没有选择空间的实现细节，通常不进入设计树。MCTS-Mem 的规模取决于重要设计选择的数量，而不是提交次数。

提交记录也未必写全理由。代码差异可以证明改动发生过，却未必说明原因。找到 RFC、issue、讨论或作者陈述之前，这些空白先以 `(uncertain)` 条目记录。

因此，重建工作可以逐步完成。即使某些旧选择还缺少解释，设计树仍能通过结构检查。

## 8. 阅读和维护设计树

`mcts-mem-use` 技能负责后续的查阅和维护工作。

开始一项较复杂的改动之前，先阅读相关的当前节点及其备选节点。这样可以看到项目已经试过哪些办法，以及项目为什么选择了当前方案。

项目改换方案后，应在同一组代码变更中更新设计树。

```text
修改之前          阅读当前节点及其 .alt/ 历史
修改期间          实现、测试、基准评测和评审
改换方案之后      写入新的 Fact 和成对的 Move
完成之前          运行 mcts-mem lint
```

命令行工具提供了几种查看方式：

```sh
npx mcts-mem view mcts_mem --alt --depth 2
npx mcts-mem show exec-engine mcts_mem
npx mcts-mem uncertain mcts_mem
npx mcts-mem lint mcts_mem
npx mcts-mem serve mcts_mem
```

`view` 输出设计树概览，`--alt` 会同时显示被否决的备选方案。

`show` 显示一项设计选择，以及它的 Facts 和 Moves。

`uncertain` 列出其理由仍缺少证据支持的设计选择。

`lint` 检查格式和内部一致性。

`serve` 启动交互式浏览界面，默认地址是 `http://localhost:4173`。

两个技能位于：

```text
skills/mcts-mem-build/SKILL.md
skills/mcts-mem-use/SKILL.md
```

它们采用开放的 [Agent Skills](https://agentskills.io) 格式，可以复制到智能体的技能目录中。

## 9. 设计记忆可以用于什么

最直接的用途是让人或智能体不用从头还原历史，就能理解项目为何这样设计。结构化的设计记忆还可以支持下面几类工作。

### 9.1 重写和移植

重写的目标通常是更换实现，同时保持行为不变。但如果旧限制已经无法从代码中看出来，重写也可能重新采用以前遇到过问题的方案。

设计树记录了项目曾投入成本进行验证的方案：试过什么、为什么失败，以及重写时仍受哪些事实约束。

### 9.2 大规模重构

修改一个子系统之前，工程师可以先查看对应节点和备选方案。

这样可以避免重新采用某种局部看起来可行、但过去已证明无法满足系统整体约束的方案。

### 9.3 重新采用旧方案

被否决的方案不必永远弃用。

假设 V8 当初因为二进制体积过大而落选。产品后来从移动设备迁移到服务器，这项限制可能不再适用。

V8 备选节点仍保留在设计树中，并记录了当初放弃它的原因。团队因此可以根据新的条件重新评估这个方案，而不必从头还原整段历史。

### 9.4 由智能体执行设计搜索

一个更长期的研究方向是让智能体自行完成更多设计搜索步骤：

1. 选择一个尚未解决的设计分支；
2. 实现一种备选方案；
3. 运行测试或基准评测；
4. 评估结果；
5. 把新事实写入设计树。

MCTS-Mem 目前不会自动完成整个过程。它提供的是这个过程所需的记忆。

## 10. 哪些项目适合使用 MCTS-Mem

以下项目更可能从 MCTS-Mem 中受益：

- 长期维护、经历过多代架构的系统；
- 需要用测量结果来做设计选择的性能敏感项目；
- 反复重写或移植到不同平台的项目；
- 人员流动时容易丢失关键背景的团队所维护的项目；
- 由智能体维护，且当前代码没有写明智能体所需设计背景的项目。

对于一次性原型，或完全遵循一份完整、稳定规范的小项目，通常没有必要建立这棵树。

主要风险是设计树与代码逐渐偏离。

如果代码持续变化，设计树却不再更新，读者可能会把已经不适用于当前系统的解释当成事实。过时的结构化记忆因为条理清楚，反而可能比明显不完整的零散笔记更容易让人信以为真。

实际维护规则很简单：每次改换方案，都在实现它的同一组变更中更新设计树，并在评审期间运行 linter。

## 11. 总结

代码保存最终采用的设计，Git 保存代码变化的过程，文档保存其中一部分解释。

MCTS-Mem 关注把三者联系起来的设计探索过程：当前方案、被否决的备选项，以及探索过程中得到的证据。

它的格式刻意保持简单，只包含 Markdown 文件、目录结构、来源标记、成对的 Moves 和一个 linter。它不试图收录项目中的全部知识，只让重要设计选择背后的理由能够长期保存、查阅和检查。

关于整体设计和 MCTS 类比的适用边界，参见 [`rationale.md`](rationale.md)。
